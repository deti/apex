Azure Load Testing is Microsoft's managed load-testing service, GA since 2022. It sits alongside Application Insights and Azure Monitor as the performance-testing leg of the Azure observability stack. The model is "bring your own JMeter/Locust script, we handle infrastructure".

## What it runs

Two script frameworks are natively supported:
- **Apache JMeter** — upload a `.jmx` file plus any plugin JARs and property files. The full JMeter plugin catalog is supported; Azure documents guidance for plugins.
- **Locust** — upload a `locustfile.py` plus requirements. Third-party Locust plugins and Python libraries are supported.

Additionally a **Quick Test** mode generates a URL-based HTTP test from just an endpoint URL — no script authoring required, Azure constructs a JMeter plan behind the scenes.

Protocol coverage is whatever JMeter or Locust support, which for JMeter means HTTP, HTTPS, REST, JDBC, TCP, JMS, MQTT, FTP, LDAP, etc.

## Scale architecture

Azure Load Testing provisions "test engines" (load generator containers) on demand. You set the engine count in the test configuration; each engine runs the uploaded script simulating a share of total virtual users. Maximum scale is thousands of concurrent users per engine, tens of engines per test. Engines run inside Azure but can be injected into a VNet to reach private endpoints and on-prem targets.

## Azure Monitor integration

The killer feature. When the SUT is Azure-hosted, Azure Load Testing automatically correlates client-side load-test metrics with server-side metrics from Application Insights, Container Insights, and Azure Monitor — CPU, memory, DB read/write rates, container resource consumption, HTTP response codes. The dashboard shows both sides of the wall during the test. No manual metric plumbing.

## Pass/fail criteria

Test configuration specifies "fail criteria": per-metric thresholds on average response time, error rate, and custom JMeter metrics. Breaching a criterion marks the test failed and (optionally) auto-stops the run. This is the CI gate.

**Auto-stop** is notable: Azure Load Testing automatically halts tests if error rates spike, preventing runaway costs from misconfigured tests. Users can also configure JMeter's AutoStop listener.

## CI/CD integration

Native support for **Azure Pipelines** and **GitHub Actions** via a first-party task/action. Trigger runs on PRs, fail the pipeline on SLO violation, attach the results URL to the PR comment. Azure CLI (`az load test run`) covers any other CI system.

## Data residency

In-region data residency: test scripts, configurations, and results never leave the Azure region where the load-testing resource is deployed. Encryption with service-managed keys by default; customer-managed keys optional. This is a compliance selling point.

## Comparison to BlazeMeter

Both target the "JMeter at cloud scale" market. BlazeMeter has broader framework support (Gatling, k6, Selenium, Playwright) and richer historical trend dashboards; Azure Load Testing has tighter Azure integration, simpler pricing, in-region data residency, and zero-setup server-side metrics correlation. Teams already committed to Azure tend to prefer Azure Load Testing; multi-cloud shops stay with BlazeMeter.

## Failure modes

- **No native support for Gatling, k6, or custom tools** — JMeter and Locust only. Azure explicitly does not support other frameworks.
- **JMeter GUI mode discouragement still applies** — the uploaded `.jmx` must be authored locally; Azure only runs non-GUI mode.
- **VNet injection complexity** — reaching private endpoints requires VNet configuration that is not trivial to set up.
- **Cost tracking is per-engine-hour** — long tests across many engines surprise people.
- **Locust support is newer than JMeter support** — feature parity is still catching up.

## Relevance to APEX G-46

Azure Load Testing's fail-criteria + auto-stop mechanism is a close analogue of what G-46's SLO assertion + resource-exhaustion-detection pipeline aims for. The "upload JMeter, run at scale, gate PRs" workflow is exactly where APEX-discovered worst-case inputs would land in an Azure-committed enterprise: APEX generates the `.jmx`, Azure Load Testing runs it, PRs gate on the results.
