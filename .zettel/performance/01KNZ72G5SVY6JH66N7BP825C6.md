A practical pattern: use the **older version** of the software as the oracle for the newer version. Rather than asking "is this latency acceptable?" you ask "is this latency acceptable *relative to what we had yesterday*?". In Barr et al. 2015's taxonomy, this is a **derived oracle** — specifically a regression oracle (vs old version) or a differential oracle (vs a reference implementation). It is by far the most common performance oracle in production CI/CD systems because it is cheap, requires no specification, and survives baseline drift (if you refresh the baseline periodically).

## The pattern in its purest form

Two deployments, one baseline (current production or current main), one candidate (proposed change). Run the same workload against both on the same hardware. Compare some performance metric. Promote the candidate if it is not significantly worse; roll back otherwise.

Implementations span the CI/CD spectrum:

- **Microbenchmark A/B**: `git bisect` runs two builds of the same benchmark, compares deltas. Criterion.rs's `--save-baseline` / `--baseline` flags are designed for exactly this.
- **Canary analysis**: Kayenta, Argo Rollouts, Flagger, Spinnaker — the baseline is a freshly-deployed instance of the current version, the candidate is a freshly-deployed instance of the new version, both receive a traffic slice.
- **Chrome Pinpoint**: bisection is iterated A/B comparisons — at each step, compare two adjacent builds under controlled hardware.
- **Perf try bots**: developer submits a patch to a try server that runs the benchmark on dedicated hardware and reports the delta against the pre-patch build.
- **Production shadow traffic**: mirror real traffic to a candidate instance, compare per-request latencies, aggregate. Used at Google, Facebook, and others.

## Why A/B beats absolute thresholds as an oracle

1. **No absolute spec needed.** You don't have to know "the right" latency; you only need to know whether your new version is *worse*. For many systems, the right latency is unknowable or changes constantly — but "not worse than yesterday" is a concrete, maintainable target.
2. **Hardware-independence.** If both A and B run on the same machine, machine-specific noise (Mytkowicz, cache state, thermal) largely cancels out in the comparison. This is the single most valuable property for CI pipelines on shared hardware.
3. **Drift-tolerant.** As long as you refresh the baseline (e.g., nightly against current main), slow improvements and slow degradations in the baseline don't require recalibration — the A/B comparison is always fresh.
4. **Composes with statistical tests.** The underlying comparison can be a t-test, Mann-Whitney, or a change-point detector; all of them fit naturally on top of an A/B pair.

## Where A/B fails

### 1. Catches deltas, not absolute badness

The oracle says "not worse than before". If "before" was already terrible, A/B gives it a pass. A codebase that has been silently leaking 0.1% per release for three years accumulates a 30% regression that never triggered any A/B gate. The fix is to pair A/B with occasional absolute SLO checks — the differential oracle is a ratchet but you need an anchor.

### 2. Dependent on baseline quality

If the baseline is noisy, the A/B comparison is noisy. If the baseline is on a different machine than the candidate, hardware heterogeneity bleeds in. Clean A/B requires controlled conditions for both legs, which is why Netflix Kayenta deploys fresh baselines (not production) and Chrome Pinpoint uses dedicated lab machines.

### 3. Interaction effects

Some regressions only appear under production load patterns that no synthetic benchmark reproduces. A/B on a benchmark suite misses these. Canary analysis with production traffic handles it; pre-merge A/B on synthetic benchmarks does not. Most teams combine both.

### 4. Can miss shared regressions

A bug that slows both A and B equally is invisible to differential testing. This is the metamorphic-testing "shared bug" problem restated. For performance, a shared regression can happen if (a) a common library is patched in between the A and B builds, (b) the test infrastructure changes under both, (c) the OS/kernel gets updated.

### 5. Ratchet problem — regressions that sneak in below the gate

If your A/B gate is "reject if candidate is > 5% slower", you can sneak in a 4.9% regression per release forever. After 10 releases, you're 40% slower. Some teams mitigate this with rolling baselines that average over a longer window, so a series of sub-threshold regressions eventually exceeds the sum threshold. Others require explicit approval for any regression, however small.

### 6. Fresh-vs-warm bias

Performance of a fresh deployment (cold caches, cold JIT, unfilled connection pools) differs from steady-state performance. Comparing a fresh canary against a warm production instance produces a systematic bias — the canary looks worse because it's fresh, not because it has a regression. Kayenta's design decision to compare against a **freshly-deployed baseline** (not production) is the standard fix; if your A/B pair doesn't match warmup state, the oracle is lying.

## Design decisions that matter

For any team building A/B perf gating, the key decisions:

1. **How many iterations per leg?** More iterations = tighter variance estimates but more CI cost. The MongoDB / Daly approach: run until statistical significance is reached, variable count. Criterion.rs default: 100 iterations. Kayenta: determined by the duration of the canary period.
2. **Paired or unpaired?** If you can run A and B interleaved (same machine, alternating), you can use paired statistics (Wilcoxon signed-rank) which is more powerful than unpaired (Mann-Whitney). Most CI pipelines run A and B back-to-back, not interleaved, losing this advantage.
3. **Warmup policy.** Discard first-N iterations on both legs, same N, same criteria. Asymmetric warmup is a hidden source of bias.
4. **Hardware sameness.** A/B on the same machine is ideal. A/B across machines, even of the same SKU, has residual differences (thermal state, uptime, filesystem layout). Same-machine A/B requires serialisation, which doubles CI latency.
5. **Baseline refresh cadence.** Daily, per-merge, or on-demand. Too stale and it stops reflecting "current main". Too fresh and you lose long-term drift detection.

## Relation to Barr et al. taxonomy

In the 2015 oracle taxonomy:
- **A/B against a previous version** = regression oracle.
- **A/B against a reference implementation** (e.g., a slower but known-correct Python reference vs an optimised Rust port) = N-version / differential oracle.
- **A/B against a fresh baseline of the same version** (Kayenta) = regression oracle with controls on baseline freshness.
- **A/B with production shadow traffic** = regression oracle with realistic workload.

All are "derived oracles" in Barr's terminology — they derive the expected behaviour from another execution, not from a specification.

## Adversarial takeaway

A/B testing is the workhorse of production perf gating because it's cheap and pragmatic, but it is structurally unable to detect absolute regressions and is susceptible to sub-threshold creep. A complete CI/CD perf strategy combines A/B (fast feedback, drift-tolerant) with periodic absolute checks (SLO assertions, complexity PMRs, or golden-reference comparisons) to anchor the ratchet.
