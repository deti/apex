# CVE-2022-25883: semver (Node.js) ReDoS in new Range

## Metadata (from NVD)

- **CVE ID:** CVE-2022-25883
- **Published:** 2023-06-21
- **Last modified:** 2025-09-23
- **CVSS v3.1 (NIST):** 7.5 HIGH — `AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:H`
- **CVSS v3.1 (Snyk):** 5.3 MEDIUM — `AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L`
- **Weakness:** CWE-1333 (Inefficient Regular Expression Complexity)

## Official description (NVD, verbatim)

"Versions of the package semver before 7.5.2 are vulnerable to Regular Expression Denial of Service (ReDoS) via the function new Range, when untrusted user data is provided as a range."

## Affected versions

- `semver` < 5.7.2
- `semver` ≥ 6.0.0 and < 6.3.1
- `semver` ≥ 7.0.0 and < 7.5.3 (NVD; Snyk lists the first fixed version as 7.5.2, reflecting the initial partial patch)

## Why it matters

`semver` is one of the most-depended-upon packages on npm (north of 100M downloads per week at the time of disclosure), vendored transitively into the core of almost every Node.js ecosystem tool: `npm` itself, Webpack, Rollup, ESLint, Yarn, the JavaScript test runners, and countless build-time tooling pipelines. A ReDoS in `new Range` means any service that accepts a semver range string from an untrusted source — package registries, CI entrypoints, SaaS services that let users paste version constraints, etc. — can be stalled with a crafted range.

## Root cause

The vulnerability is in the regular expression used to parse range expressions. The pattern matches a version range of the form `<operator><version>` repeated, and uses a whitespace+operator construct that produces overlapping matches when the input contains long runs of pathological separators. The resulting NFA has two paths that consume the same characters, triggering super-linear backtracking on failure. The classic ReDoS family is: an alternation or nested quantifier that can match the same substring multiple ways, followed by a mandatory character the attacker omits to force the engine to walk every partition.

The fix (upstream commit `717534ee353682f3bcf33e60a8af4292626d4441`, GitHub PR #564) rewrites the range grammar to avoid the overlap and adds a hard cap on range string length.

## Exploitation

A proof-of-concept (released by Snyk and referenced from the GitHub advisory) supplies a long string of characters that look like operators but never satisfy the full range grammar, e.g. many spaces followed by an invalid trailing character, causing the `new Range(input)` call to take multiple seconds of CPU for a few kilobytes of input. On a typical Node.js server with a single event loop, this is sufficient to stall the process entirely.

## Remediation

- Upgrade to `semver >= 7.5.2` (branch 7), `semver >= 6.3.1` (branch 6), or `semver >= 5.7.2` (branch 5).
- If upgrade is impossible, pre-validate the length and character set of any untrusted range string before passing it to `new Range`, e.g. reject any input longer than a few hundred bytes or containing repeated whitespace runs.
- At the platform level, never call `semver.Range` on user-supplied data inside a request handler without a timeout guard.

## Relevance to APEX G-46

`semver` is a textbook G-46 target: small pure function, tainted string input, cheap test harness. A correct APEX performance generator should find this regression within seconds using a PerfFuzz-style max-per-edge counter applied to the regex engine. It is an excellent concrete CVE to include in the regression suite, and pairs naturally with `CVE-2023-26115 (word-wrap)` and `CVE-2020-5243 (uap-core)` as a trio of Node.js ReDoS CVEs covering the three main catastrophic-backtracking patterns (whitespace runs, alternation overlap, and user-agent fingerprint overlap respectively).
