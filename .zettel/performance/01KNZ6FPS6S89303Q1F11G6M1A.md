*Source: Charlie Curtsinger, Emery D. Berger — "STABILIZER: Statistically Sound Performance Evaluation" — ASPLOS 2013. ACM DOI 10.1145/2451116.2451141. Open-source at github.com/ccurtsinger/stabilizer.*

Stabilizer is the operational response to the problem diagnosed by Mytkowicz et al. 2009. Mytkowicz showed that layout accidents — link order, environment variable length, filesystem path — produce systematic performance differences that can dwarf real effects. Stabilizer's thesis: instead of holding layout constant (impossible) or accepting the bias (unscientific), **continuously re-randomize layout at runtime**, so that layout effects become Gaussian noise that can be handled with standard statistical tests (t-tests, ANOVA).

## The thesis

> "Stabilizer is a system that enables the use of the powerful statistical techniques required for sound performance evaluation on modern architectures. Stabilizer forces executions to sample the space of memory configurations by repeatedly rerandomizing layouts of code, stack, and heap objects at runtime."

Importantly:

> "Re-randomization also ensures that layout effects follow a Gaussian distribution, enabling the use of statistical tests like ANOVA."

This is the key insight that makes Stabilizer interesting beyond "also randomize things": by **repeatedly** re-randomizing during execution (not just once at startup), the system averages the execution over a distribution of layouts, and the central limit theorem does the rest. You end up with a measured runtime that is approximately Gaussian around the "true" cost of the computation, with layout noise as the variance term.

## Technical approach

Stabilizer is implemented as an LLVM pass plus a runtime library:

1. **Code randomization.** Functions are relocated at program startup and periodically during execution. The LLVM pass inserts trampolines so that call sites can resolve through a table; the runtime updates the table when functions move. Rewriting happens every ~500ms.
2. **Stack randomization.** The stack pointer is offset by a variable amount so that stack-allocated variables land at different addresses on each function call. This randomizes conflict misses in L1-D.
3. **Heap randomization.** A randomizing allocator (akin to DieHard) places heap objects at random addresses within a large virtual address range, so that cache-line alignment and TLB behaviour vary randomly across runs.

## Results

- **Overhead: less than 7% median.** This is the surprising result — continuous randomization sounds expensive but is actually cheap because the randomization interval is large relative to instruction cost.
- **Evaluation on SPEC CPU2006 and PARSEC benchmarks.**
- **Key finding applied: the performance difference between `-O2` and `-O3` is statistically indistinguishable from noise.** Under Stabilizer's randomized layout, -O3 does not reliably beat -O2 across a range of benchmarks. This is a direct challenge to the folklore that "-O3 is always faster."
- **`-O2` vs `-O1`, by contrast, shows a clear statistical separation** — the gap is large enough to survive layout randomization.

## Why this matters

Before Stabilizer, compiler and systems researchers generally reported benchmark numbers without bias controls. Reviewers accepted "we ran each benchmark 10 times and took the mean" as rigorous. Stabilizer is a direct demonstration that this standard methodology **reports wrong conclusions**: you get statistically significant differences that are actually layout artifacts, not real effects. After Stabilizer, the rigorous bar rose.

Stabilizer matters for CI perf gating because:
- **Wall-clock benchmarks on shared CI runners are measurement-biased.** Each commit has a fresh link, possibly a different working directory, possibly a different machine from the runner pool. Differences across runs are a mixture of real regressions and layout accidents.
- **Without randomization, you can't tell them apart.** Stabilizer provides one answer (randomize in-process); instruction counting (CodSpeed/iai-callgrind) provides another (sidestep wall-clock entirely); bare-metal perf labs (Google Perflab) provide a third (tightly control layout).
- **It also implies that tools like Criterion.rs and Google Benchmark, which use t-tests or bootstrap methods but do not randomize layout, are subject to the same bias** when run on typical hardware. Their statistical rigor is conditional on layout being controlled by other means.

## Limitations

- **Not integrated with LLVM upstream.** Stabilizer is a research prototype; keeping it working against newer LLVM versions requires effort. Later work (Polish & Reed) has tried to modernize it but adoption remains limited.
- **Language-specific.** The original Stabilizer targets C/C++ through LLVM. Randomizing JIT-compiled code in managed languages (JVM, .NET) requires separate work — Curtsinger has proposed extensions but they are less mature.
- **Randomization does not help with all noise sources.** Thermal throttling, background processes, turbo frequency variation, and network jitter are orthogonal — Stabilizer addresses layout, not the rest of the noise budget.
- **Re-randomization interval is a tunable knob.** Too short and you pay overhead for marginal variance reduction; too long and the central-limit argument weakens. The paper picks ~500ms empirically.
- **Assumes Gaussian layout effects.** The paper argues this empirically but it's not a theorem. For benchmarks dominated by a single layout-sensitive code path, the distribution can be multimodal even under randomization.

## Adversarial commentary for CI perf gating

Stabilizer is not widely deployed in production CI pipelines. Why?
- **LLVM integration cost** — the pass has to be maintained.
- **Not a drop-in solution** — you have to rebuild the benchmark binary with the Stabilizer pass.
- **Wall-clock CI runners still dominate**, and they have problems Stabilizer doesn't solve (thermal, turbo, noisy neighbours).
- **Instruction-count benchmarking (CodSpeed)** bypasses the entire problem more cheaply for a large class of CPU-bound benchmarks.

The right conceptual framing is: Stabilizer proves the problem is real and gives one principled solution. Most production teams learn from the proof and pick a cheaper but less principled mitigation (CodSpeed, perflab, or just "accept the noise and use change-point detection").
