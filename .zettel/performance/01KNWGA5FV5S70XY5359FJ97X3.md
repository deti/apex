# OWASP: Regular Expression Denial of Service (ReDoS)

*Source: https://owasp.org/www-community/attacks/Regular_expression_Denial_of_Service_-_ReDoS — fetched 2026-04-10.*

## Introduction

OWASP defines ReDoS as "a Denial of Service attack, that exploits the fact that most Regular Expression implementations may reach extreme situations that cause them to work very slowly (exponentially related to input size)."

## Attack Mechanism

### The Core Problem

The vulnerability stems from how naive regex engines work. These engines construct a Nondeterministic Finite Automaton (NFA) that may have multiple possible paths for a given input. When matching fails, the engine uses backtracking — returning to previous decision points to try alternative paths. Problematic patterns force the engine to explore exponentially many paths.

The page illustrates this with the pattern `^(a+)+$`. For input `aaaaX`, there are 16 possible paths; for `aaaaaaaaaaaaaaaaX`, there are 65,536 paths — doubling with each additional character.

### Evil Regex Characteristics

Patterns become vulnerable when they contain:
- Grouping with repetition
- Inside repeated groups: repetition or alternating patterns with overlapping matches

Examples provided include `(a+)+$`, `([a-zA-Z]+)*$`, `(a|aa)+$`, and `(a|a?)+$` — all susceptible to input like `aaaaaaaaaaaaaaaaaaaaaaaa!`.

## Real-World Examples

**Email Validation (ReGexLib ID 1757):**
The vulnerable portion: `(([\-.]|[_]+)?([a-zA-Z0-9]+))*` paired with input `aaaaaaaaaaaaaaaaaaaaaaaa!`

**OWASP Repository Java Classname Pattern:**
The problematic section: `(([a-z])+.)+` with malicious input `aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa!`

## Exploitation Scenarios

The page describes a code-injection attack where user input becomes a regex pattern itself. An example shows username validation code that constructs regex dynamically without sanitisation, allowing attackers to inject evil patterns directly.

## Risk Context

The document emphasises that regex patterns exist throughout web infrastructure — browsers, WAFs, databases, and servers — making ReDoS a widespread threat across all application layers.

## Relevance to APEX G-46

OWASP's characterisation of evil-regex structural patterns (grouping + repetition + overlap) is exactly the pattern signature a static ReDoS detector should look for. The "Exploitation Scenarios" section adds a second concern: APEX should flag not only *literal* regex patterns that are vulnerable, but also *dynamically constructed* regex where user input flows into the pattern itself. This is a taint-analysis problem on top of the pattern-matching problem — it should be handled by APEX's CPG-based taint engine (apex-cpg) in concert with the ReDoS detector.
