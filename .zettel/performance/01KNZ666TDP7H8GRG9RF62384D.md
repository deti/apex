OpenTelemetry (OTel) is the CNCF-graduated open standard for observability instrumentation. It unifies the previously fragmented worlds of OpenTracing and OpenCensus into a single specification, SDK set, and wire protocol (OTLP). For performance investigation work, the critical OTel features are **traces**, **metrics**, and — decisively — **exemplars** that link the two.

## Signals

OTel defines three primary signals:

- **Traces** — distributed causal chains of operations across services. Spans are the building blocks.
- **Metrics** — aggregated numeric measurements over time. Counters, histograms, gauges.
- **Logs** — structured log records with trace correlation via span context.

A fourth signal, **profiles**, is under active standardisation (as of the source fetch). When it lands, OTel will cover the full four-pillar model of observability: traces, metrics, logs, profiles.

## Traces

A trace represents the path of a request through a distributed system. Each unit of work is a **span** with:

- A trace_id (shared by all spans in one trace) and a span_id (unique per span).
- A parent span_id (or root if absent), establishing hierarchy.
- A name, start/end timestamps.
- **Attributes** — key/value pairs (strings, booleans, numbers, or arrays of these). Represent dimensional metadata (`http.method`, `db.system`, `user.id`).
- **Events** — timestamped sub-points within a span. Used for things like "cache miss occurred here" where the timestamp matters.
- **Links** — causal associations to other spans that are not strictly parent/child. Critical for async operations (a span finishes, enqueues a job, the job's eventual span links back to the queuer).
- **Status** — Unset (default success), Ok (explicitly successful), Error (failure).

**Span kinds** classify the span:
- `Internal` — operations wholly within a process boundary.
- `Server` — synchronous incoming remote call.
- `Client` — synchronous outgoing remote call.
- `Producer` — creates asynchronously processed jobs.
- `Consumer` — processes producer-created jobs.

**Context propagation** is the mechanism that carries trace_id + span_id + trace flags across process boundaries. HTTP uses the W3C `traceparent` and `tracestate` headers; gRPC uses metadata; messaging systems use broker-specific mechanisms. Propagation is what makes distributed tracing actually distributed — without it, you get independent per-service spans with no causal connection.

## Metrics data model

OTel metrics distinguish the **Event model** (how instrumentation reports data), the **Stream model** (what flows over OTLP), and the **Timeseries model** (what backends store).

Primary point kinds:

- **Sum** — additive. Monotonic (counters) or non-monotonic. Temporality: cumulative or delta.
- **Gauge** — last-sample semantics. No aggregation function, just the most recent value.
- **Histogram** — compressed population via explicit buckets, count, and sum.
- **ExponentialHistogram** — high-dynamic-range histogram with exponentially-spaced bucket boundaries at a configurable scale. The native latency-distribution primitive in OTel, equivalent in purpose to HdrHistogram.

Temporality:
- **Cumulative** — each point includes everything since a start timestamp. Prometheus-style.
- **Delta** — each point covers a discrete time range. StatsD-style. Enables stateless clients.

Views allow temporal re-aggregation, spatial re-aggregation (dropping attributes), and delta-to-cumulative conversion without losing semantics.

## Exemplars — the key feature for performance investigation

**Exemplars are the OTel feature that matters most for performance work.** An exemplar is a specific observation — a single request, a single allocation, a single query — that also contributes to an aggregated metric (a histogram, a sum, a gauge). Its structure includes:

- The value (latency in milliseconds, bytes allocated, etc.).
- The timestamp.
- An optional trace context: **trace_id and span_id**.
- Filtered attributes providing dimensional context.

What exemplars enable: when you look at a latency histogram and notice that p99 is spiking to 2.5 seconds, you can click through the exemplar attached to the p99 bucket and land in the actual distributed trace that produced that observation. The metric tells you *that* something is slow; the exemplar tells you *which request* is slow; the trace tells you *why* it is slow.

This is the modern observability answer to the problem Dean & Barroso described in "The Tail at Scale": when the p99 response moves, you cannot just stare at an aggregated dashboard — you need a causal link back to the *specific slow events*. Exemplars provide that link at the protocol level.

For histogram metrics, exemplar values already participate in the aggregated counts, so there is no double-counting. Sampling strategy is implementation-defined — the spec permits storing exemplars only for bucket extremes, or probabilistically, to control cardinality.

## OTLP — the wire protocol

Traces, metrics, and logs share a single protocol buffer-based wire format (OTLP) that rides over gRPC or HTTP/protobuf. This is what lets a single OTel Collector ingest data from any instrumented service and route it to multiple backends (Prometheus for metrics, Jaeger/Tempo for traces, Loki for logs) without per-service plumbing.

## Strengths

- Single standard replacing the OpenTracing/OpenCensus split.
- Vendor-neutral — every APM vendor supports OTLP ingestion.
- Exemplars link metrics and traces at the protocol level, not via out-of-band tooling.
- Context propagation standardised via W3C Trace Context.
- Language coverage is broad (20+ official SDKs).

## Failure modes

- **Cardinality explosions** on metrics — an attribute set that's per-user becomes a new timeseries per user, blowing up storage.
- **Sampling decisions at span creation** can discard traces that later turn out to be interesting. Tail-based sampling (decide after the whole trace is seen) is more expensive but often necessary.
- **Exemplar support in backends varies** — Prometheus has exemplar storage, but many older backends don't. Tempo, Grafana Mimir, and other new stores support it; legacy stores may drop exemplars on ingest.
- **The spec is large** — full correct instrumentation is non-trivial and many SDKs have subtle divergences from the spec.
- **Span-based instrumentation has performance overhead** — typically 1-5% per instrumented call site, which matters at very high RPS.

## Relevance to APEX G-46

OpenTelemetry is the observability substrate G-46's performance-regression-detection phase should consume. When APEX reports "this endpoint got 2x slower", the actionable follow-up is the OTel trace of a representative slow request — and exemplars on the latency histogram are the ready-made link to that trace. Emitting performance findings as OTLP events (metrics with exemplars pointing at the APEX-generated worst-case input trace) is the most natural integration path.
