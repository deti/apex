Perfherder is Mozilla's performance regression dashboard for Firefox. Together with Talos (the main microbenchmark suite) and the Mozilla Performance Sheriff rotation, it is one of the longest-running and most operationally-mature examples of continuous perf monitoring on a large open-source project. It predates MongoDB's Daly-style change-point detection system and uses a different (simpler) approach that has nonetheless scaled to tens of thousands of benchmarks across multiple platforms for over a decade.

## What Talos is

Talos is Firefox's performance microbenchmark suite. It runs on Mozilla's CI infrastructure (Taskcluster) against virtually every check-in to an integration branch. Measurements cover page load, scrolling, startup, memory, and JavaScript microbenchmarks, across Windows, macOS, Linux, and Android. Talos is dedicated: Mozilla runs it on a **dedicated hardware pool** (not the general CI pool) to reduce noise from neighbours, background services, and machine heterogeneity. This is the "bare-metal perf lab" answer to the CI noise problem.

## How Perfherder detects regressions

From the Firefox source docs:

> "Alerts are generated by programmatically verifying there is a sustained regression over time (original data point + 12 future data points). Perfherder uses a T-test to detect significant changes by comparing to a fixed threshold, and if the change magnitude surpasses a certain threshold, an alert is triggered."

Key points:

1. **Post-hoc detection with lag.** Perfherder does not alert at commit time; it waits for 12 additional data points (i.e., 12 subsequent test runs on that benchmark) to confirm the regression is sustained. This is a deliberate trade-off: the latency of detection is ~12 runs, but the false-positive rate is dramatically lower than pairwise comparison.
2. **T-test + threshold.** A t-test is used to assess whether the post-change window is statistically different from the pre-change window. Combined with a **fixed threshold** on the change magnitude — both significance and effect-size gate must fire. This is a simple, effective design that is still inferior to non-parametric change-point detection (Daly et al. 2020) for heavy-tailed or multimodal benchmarks but works well on Talos's relatively stable dedicated-hardware runs.
3. **Alert grouping.** Alerts related to the same revision (commit) are grouped into an **alert summary**. A sheriff sees one grouped entry per revision, not twelve individual alerts for a single regression that hit twelve benchmarks.

## The Performance Sheriff rotation

Mozilla runs a dedicated rotation of performance sheriffs who triage alerts:

> "Performance Sheriffs look at the data produced by performance test jobs (mainly Talos), find regressions, determine their root cause(s) and get bugs on file to track all issues."

The sheriff's workflow, as documented:
1. Review the alert summary dashboard.
2. For each alert, examine the graph to confirm a real shift (not a flaky benchmark).
3. Use Talos's revision-bisection tools to narrow down the responsible commit within the change range.
4. File a bug on Bugzilla assigned to the commit author.
5. Optionally run **perf try bots** (see below) to verify a proposed fix.
6. Sign off on the alert as actioned.

This is exactly the same operational pattern as MongoDB's sheriff rotation (Daly 2021) — the algorithm is different but the organisational structure is nearly identical. Both teams independently concluded that **a human in the loop is required for actionable perf regression detection at scale**.

## Perf try bots

Mozilla provides a "try" server where developers can submit pre-merge changes to run perf tests before landing. Try bots run Talos against the proposed patch on dedicated hardware and return results the developer can inspect. This is the pre-merge complement to post-merge detection.

From the Chromium docs (same pattern):

> "It's best to run a perf tryjob, since the machines in the lab are set up to match the device and software configs as the perf waterfall, making the regression more likely to reproduce."

## Comparison to Daly et al. 2020

Both Mozilla (Perfherder/Talos) and MongoDB solve the same problem with different tools:

| | Mozilla (Perfherder) | MongoDB (Daly 2020) |
|---|---|---|
| Algorithm | t-test on pre/post windows | E-divisive means change-point |
| Noise control | Dedicated hardware (Talos farm) | Dedicated hardware (Evergreen) + statistical robustness |
| Detection latency | 12 runs (hours-days) | Typically a few runs |
| Triage | Human sheriff rotation | Human sheriff rotation |
| Assumes Gaussian? | Yes (t-test) | No (non-parametric E-divisive) |
| Open source | Yes (Treeherder / Perfherder code) | Partially (signal-processing-algorithms) |

Mozilla's approach works because the dedicated hardware gives Gaussian-enough noise for the t-test to be valid; MongoDB's approach is more general because it handles cases where the noise is not well-behaved. Both emphasise the sheriff rotation as the make-or-break organisational element.

## The Perfherder dataset

A 2025 paper by Besbes et al. (arxiv.org/abs/2503.16332) released a **dataset** of Firefox performance measurements and Perfherder alerts:
- 5,655 performance time series
- 17,989 performance alerts
- One year coverage (May 2023 – May 2024)
- Detailed bug annotations and testing metadata

This is one of the few public datasets of real CI perf measurements plus labelled ground-truth regressions, which is a significant resource for researchers building new change-point detection or ML-based regression detection methods. Hosted on Zenodo, DOI 10.5281/zenodo.14642238.

## Adversarial commentary

- **T-test on dedicated hardware is fine until it isn't.** If Mozilla ever had to run Talos on shared hardware, the t-test's Gaussian assumption would break and the alert volume would explode. The entire pipeline depends on the hardware pool being quiet.
- **12-point confirmation window is slow.** For fast-moving regressions, humans notice degraded browser performance before Perfherder emits an alert. The confirmation window trades false positives against detection latency, and Mozilla has chosen low false positives.
- **Fixed magnitude threshold doesn't scale with benchmark noise.** Some Talos benchmarks are inherently noisier than others; a single global threshold either over-fires on noisy ones or under-fires on quiet ones. Perfherder has per-benchmark threshold overrides but setting them is manual.
- **No effect-size output.** Alerts report magnitude as a percentage but don't include Cliff's delta or similar non-parametric effect sizes.
- **Alert fatigue is an ongoing operational concern.** Sheriffs report cycles of "alert volume too high → raise threshold → miss real regressions → lower threshold → alert volume too high." The same treadmill as any threshold-based alerting system.
