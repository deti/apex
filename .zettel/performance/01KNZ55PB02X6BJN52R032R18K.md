# AsyncAPI — Spec-Driven Testing for Event-Driven Architectures

AsyncAPI is to event-driven APIs (Kafka, AMQP, MQTT, WebSocket, Solace) what OpenAPI is to HTTP REST. An AsyncAPI document describes which channels (topics) a service publishes to and subscribes from, and the schema of messages on each channel. The spec has been moving through v2 and v3 drafts since 2017.

## Why it matters for this research

Event-driven workloads are the biggest blind spot in load-test generation. The HTTP-centric ecosystem (k6, Gatling, JMeter) treats Kafka and friends as an afterthought. In practice, a lot of real production performance incidents happen in event-driven paths — slow consumers, lag, rebalance storms — and the schema-driven generators for them are immature.

## Tooling landscape

### Microcks

Microcks is an open-source API mocking and contract-testing platform (Red Hat / CNCF sandbox). It ingests OpenAPI, AsyncAPI, Postman, Pact, and WSDL. For AsyncAPI specifically, Microcks can *connect to a real Kafka broker* and start producing or consuming messages that conform to the AsyncAPI schema — effectively turning the spec into a live event producer for downstream testing.

Key capabilities:

- Schema-conformant message generation on Kafka/MQTT/AMQP/WebSocket topics.
- Contract testing mode: validates that messages observed on a topic conform to the AsyncAPI schema.
- No native load-profile configuration — it publishes at whatever rate the mock harness drives.

### kloadgen (JMeter plugin)

kloadgen is a JMeter plugin for Kafka that supports schema registry, Avro, Protobuf, and JSON Schema. It is the most commonly cited path to produce a *load* test against Kafka today. It is workload-profile-aware because it inherits JMeter's thread group and timer primitives.

Failure modes: JMeter's UI is heavy, the Avro-with-schema-registry path has a lot of configuration, and the resulting JMX files are not readable enough to maintain cleanly.

### Specmatic

Specmatic is a contract-testing tool that supports AsyncAPI with an overlay mechanism for "triggers and side-effects": a test case can specify "when event X is published to topic A, the system should publish event Y to topic B." This is the closest thing in the AsyncAPI space to RESTler's producer-consumer modelling, and it's more natural for event-driven because causality is the whole point.

### Solace Event Feeds

Solace ships "Event Feeds" — a closed-source but free tool that generates production-like event streams from an AsyncAPI spec. Useful demo material; not a load generator.

## Schema-driven → load test gaps

1. **Throughput, not just correctness.** Microcks and Specmatic are correctness/contract tools; they do not natively give you "100 K events/s with field X drawn from distribution Y." kloadgen gives you throughput but very little distribution control.
2. **Consumer lag is the real signal.** For an event system, the performance metric that matters is consumer lag and end-to-end latency from produce to consume. Generators that only publish miss half the picture. You need a generator + a measured consumer harness that shares the clock.
3. **Key-distribution effects.** Kafka partitioning is key-hash-based; skew in the key distribution causes partition hot spots that linearly degrade throughput. None of the AsyncAPI-driven generators let you specify the key-value distribution realistically.
4. **Header-based routing / filtering effects.** Headers drive consumer-group filtering in some systems. Generators rarely touch headers.
5. **Backpressure coupling.** The producer's throughput depends on the consumer's behaviour (ack latency, flow control). Realistic tests need both sides controllable.

## Toolmaker gap

There is no "k6 for Kafka" — no single engineer-friendly tool that takes an AsyncAPI spec and produces a runnable, tunable, schema-aware event load test with headers, keys, and realistic distributions. This is a high-value open-source opportunity.

A plausible architecture: AsyncAPI → code generator emitting a JavaScript/TypeScript producer harness using the existing Kafka client libraries, with a k6-style `scenarios` block for rate control and a Microcks-style schema validator on the consumer side.
